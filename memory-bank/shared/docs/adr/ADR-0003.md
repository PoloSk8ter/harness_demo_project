# ADR-0003 — Route to exactly one specialist; AgentTool call-and-return, no merge

- **Status:** accepted
- **Date:** 2026-06-25
- **Architecture version:** v1

## Context
The PO locked: "one message routes to exactly ONE specialist; the finalizer formats that single reply and
does NOT merge." ADK offers two ways to attach a specialist: `AgentTool(agent=…)` (call-and-return, the
caller keeps control) and `sub_agents=[…]` (full delegation — the LLM can `transfer_to_agent`). The design
must make "exactly one" legible and keep the guardrail re-check (ADR-0002) in control of the exit.

## Decision
The triage agent (Seam B) emits a single `RouteDecision` label into state. The router invokes exactly the
one matching specialist as `AgentTool(agent=…, skip_summarization=True)` — call-and-return, so the
orchestrator stays in control and drives on to the synthesizer, which formats that single output and never
merges. Guardrail/triage sub-agents set `disallow_transfer_to_parent/peers=True` so the LLM cannot hand
control sideways. Routing is observable in tests via `.tool_calls()` (exactly one specialist tool fired).

## Alternatives rejected
- **`sub_agents` full delegation / transfer** — hands the whole turn to a specialist, making it harder to
  guarantee "exactly one" and to keep the guardrail re-check in control of the exit. Reserved for a future
  true dispatcher if compound multi-domain messages ever become a goal.
- **Fan-out to multiple specialists + a merging synthesizer** — explicitly rejected by the PO (more power,
  more coordination failure surface, not needed for the MVP).

## Consequences
Single-hop, traceable control flow; routing accuracy is an observable signal (one specialist tool per turn),
which ADR-0004's evals assert on. Supporting genuinely compound multi-domain messages later would reopen
this ADR (a change-request to the SA).
