# ADR-0002 — Safety is structural, not model-dependent

- **Status:** accepted
- **Date:** 2026-06-25
- **Architecture version:** v1

## Context
In this domain a refusing boundary and human oversight are non-negotiable: the CFPB holds these assistants
to the human-agent standard, and an independent audit found all 24 banking bots it tested exploitable. The
brief's case-against names the real risks of a multi-part assistant — mis-routing, dropped context, and
"a finalizer that softens a refusal" — and warns the demo could validate the easy half while leaving
adversarial robustness untouched. The PO locked three behaviours: guardrail screens first + re-checks last;
state changes are confirm-then-execute; fraud is a structured escalation. The architectural question is
*what enforces them* — the model, or the code.

## Decision
Enforce all three safety guarantees in **code at fixed seams**, never at the model's discretion:
- **Guardrail-first & recheck-last (Seam A):** screening is a fail-closed `before_model_callback`
  (`safe(on_error="raise")`) at the pipeline entry, plus a pinned `static_instruction` system guardrail
  (cannot be displaced by conversation history), plus a final re-check before return. Because screening is a
  structural gate *ahead of* the fixed `SequentialAgent`, no specialist can run on unscreened input, and the
  synthesizer cannot emit a reply the re-check refuses.
- **Confirm-then-execute (Seam C):** the mutation tool `set_card_state` refuses to change state unless a
  valid confirmation token is present in session state. The LLM drafts the action and asks; the *tool*
  enforces. Detected intent alone never writes.
- **Structured escalation (Seam D4 + E):** the fraud path emits a Pydantic `EscalationRecord`; the record is
  produced by code, not free text.

## Alternatives rejected
- **Prompt-only safety** (instruct the model to refuse / confirm / escalate) — precisely what the 24/24
  audit broke; a jailbreak or a mis-route bypasses it entirely.
- **Guardrail as a fail-OPEN after-callback** — adk_studio's conventions warn that a throwing guard wrapped
  fail-open would let the model run on unguarded input; the sanctioned form is a fail-closed before-callback.
- **Confirm-then-execute enforced only by the model asking** — a model that skips the question would
  execute; the tool-level token makes the guarantee structural rather than behavioural.

## Consequences
The demo's safety holds even under LLM mis-routing or jailbreak — directly answering the brief's
adversarial-robustness worry, and making "the pattern runs" not be mistaken for "safe to ship." The safety
mechanics become deterministically testable **without** the model (feeds ADR-0004's default CI suite). Cost:
a confirmation-token protocol and a `state_keys.py` registry. A change-request would have to overturn either
fail-closed-before-callback screening or tool-enforced writes.
